introduction

What is an ER Model?

Definition

A conceptual data model that represents the structure of a database using entities, attributes, and relationships.

Purpose

To visually map how data is organized and connected, serving as a blueprint for database design.

Key Benefit

Bridges the gap between business requirements and technical database implementation.

The ER model has evolved significantly since its introduction in 1976, adapting to new technologies and methodologies.

Peter Chen's proposal

Introduced by Dr. Peter Chen to visually model database systems.

  • First unified notation
  • Separated conceptual from implementation

building blocks

ER Components

Entity

Real-world object (CUSTOMER, ORDER).
Strong (independent) / Weak (depends).

Attribute

Properties: Simple, Composite, Key, Derived, Multi‑valued.

Relationship

Connects entities (places, includes). Has degree & cardinality.

entities

Strong & Weak Entities

  • Strong Entity – exists independently, has its own primary key (e.g., CUSTOMER identified by CustomerID).
  • Weak Entity – depends on a strong entity, no key without owner (e.g., DEPENDENT identified via Employee).

Notation: strong entity – rectangle; weak entity – double rectangle.

attributes

Attribute Types

Simple
Atomic, cannot divide (e.g., Age).
Composite
Sub‑attributes: Address = Street + City + Zip.
Key
Unique identifier (e.g., CustomerID).
Derived
Calculated from other (e.g., Age from DOB).
Multi‑valued
Multiple values (e.g., PhoneNumbers).
PK / FK
Primary Key – unique; Foreign Key – links entities.

relationship degree

Unary · Binary · Ternary · N-ary

  • Unary/Recursive – relates entity to itself (e.g., Employee manages Employee).
  • Binary – between two entities (most common: Customer places Order).
  • Ternary – three entities involved (e.g., Doctor prescribes Medicine to Patient).
  • N-ary – n entities connected.

Example: Doctor, Patient, and Medicine all connected in one ternary relationship

constraints

Cardinality & Participation

Cardinality
  • One to One (1:1)
  • One to Many (1:M)
  • Many to One (M:1)
  • Many to Many (M:N)
Participation
  • Total – every entity must participate (double line).
  • Partial – some entities may not participate (single line).

Example: 1 Department has many Employees → 1:M; every Employee belongs to a Department (total).

methodology

How to draw an ER Diagram

  1. Identify entities (rectangles).
  2. Define attributes (ovals) and mark keys.
  3. Establish relationships (diamonds).
  4. Add cardinality constraints (1, M, N).
  5. Connect with lines.

Benefits of ER Diagram
